## 1. WMS系统核心业务流程

### 题目
请描述WMS系统的四大核心业务流程，并说明入库和出库主流程的关键节点。

### 核心答案

| 流程 | 核心节点 |
|------|----------|
| **入库** | 预收货 → 实物收货 →质检 → 上架 → 确认入库 → 库存更新 |
| **出库** | 订单接收 → 挑库分配 → 拣货 → 复核 → 打包 → 交接出库 |
| **盘点** | 盘点计划 → 初盘 → 复盘 → 差异确认 → 库存调整 |
| **调拨** | 调拨申请 → 审核 → 源库位下架 → 移位/库间运输 → 目标库位上架 |

入库关键节点中，**质检**和**上架**最容易成为瓶颈；出库流程中**拣货**和**复核**是耗时最长的环节。

### 追问方向
- 入库流程中"预收货"和"实物收货"的区别是什么？预收货信息不匹配时如何处理？
- 出库订单如果库存不足，是先占库还是先通知？

### 避坑提示
- 候选人若只描述表面流程（如"入库就是扫码上架"），缺乏对质检环节、库存冻结、状态扭转的理解，说明缺乏实战经验。
- 四大流程之间的数据依赖关系（如出库依赖入库批次数据）没有提及，则对系统全局理解不足。

## 2. 库存管理常见问题

### 题目
仓库运营中常见的库存管理问题有哪些？请结合批次管理和库位管理说明原因和解决思路。

### 核心答案

| 问题类型 | 典型表现 | 根因 | 解决思路 |
|----------|----------|------|----------|
| **库存不准** | 实物与系统账不符，差异率>1% | 拣货未复核、盘点不及时、移库漏操作 | 闭环复核机制、定期盘点、条码绑定 |
| **库位混乱** | 商品放错库位、库位状态不更新 | 上架未扫码、库位映射缺失 | 库位条码化、上架强制校验 |
| **批次管理混乱** | 先入后出变后入先出、呆滞料堆积 | 未执行FIFO、缺乏效期预警 | 强制批次策略、近效期优先出库 |

### 追问方向
- 如何通过系统手段（比如库存冻结和库存日志）追溯库存不准的根因？
- 批次和序列号的区别是什么？什么场景下用批次管理、什么场景下用序列号？

### 避坑提示
- 只回答问题现象而不能说出根因和系统约束机制的，说明缺乏系统性思考。
- 候选人若将批次管理和序列号管理混为一谈，需要追问区分。

## 3. 收货流程

### 题目
收货流程中，"预收货"阶段需要做哪些操作？实物收货和质检环节的系统处理逻辑是什么？

### 核心答案

**预收货（ASN/预到货通知）**：
- 接收上游ERP/OA的采购订单或供应商交货通知
- 系统生成预收货单，记录预计数量、预计到货时间、供应商信息
- 提前校验商品信息、批次要求、包装规格

**实物收货**：
- 扫码枪扫描托盘/箱条码，系统匹配预收货单
- 清点数量，若实际数量与预收货不符，系统支持"少收/多收"处理
- 触发质检指令（可按批次抽检或全检）

**质检**：
- 合格 → 直接触发上架指令
- 不合格 → 冻结库存（不允许上架），触发退货或特采流程
- 质检结果记入商品档案，影响供应商绩效

### 追问方向
- 预收货数量和实物数量差异较大时，系统是否允许强制入库？走什么审批流？
- 质检不合格商品在系统里是什么库存状态？冻结和锁定有什么区别？

### 避坑提示
- 不清楚"预收货单"和"入库单"是两张独立单据的，说明对流程解耦理解不足。
- 收货环节缺少"差异处理"概念的，无法应对实际业务中的各种异常。

## 4. 上架策略

### 题目
WMS中的上架策略有哪些？请解释FIFO（先进先出）是如何在系统中落地执行的，以及推荐库位算法考虑哪些因素。

### 核心答案

**常见上架策略**：

| 策略 | 逻辑 |
|------|------|
| **固定库位** | 商品绑定固定库位，适合整进整出的大宗商品 |
| **随机库位** | 根据实时库位容积和拣货热度动态分配 |
| **类FIFO库位** | 同批次商品尽量集中放置在同一库区，按入库时间顺序排列 |

**FIFO系统实现**：
- 上架时记录**批次号+入库时间戳**
- 出库挑库时，系统按批次时间**升序**自动推荐库位
- 若同批次多个库位均有货，取**最早入库**库位优先
- 辅以批次校验（拣货时扫描批次码，若非目标批次则禁止出库）

**推荐库位算法考虑因素**：

1. **库位容积匹配**：商品体积 vs 库位容量
2. **拣货热度均衡**：高频商品放腰部，低频商品放高层/远端
3. **同品集中**：同SKU尽量放同区，减少拣货路径
4. **重量合规**：重货放下层，轻货放上层
5. **效期约束**：近效期货靠近出库口

### 追问方向
- 如果某批次商品效期临近，但FIFO推荐出的库位被另一个订单预占了，系统如何处理？
- 随机库位策略下，如何避免同一SKU分散在过多库位导致拣货效率下降？

### 避坑提示
- 只会背"FIFO就是先进先出"而无法解释系统如何通过批次号和时间戳落地执行的，说明缺乏系统设计理解。
- 候选人若不考虑库位容积和重量因素的，说明对仓库物理约束认知不足。

## 5. 拣货策略

### 题目
请解释订单波次的概念，以及"摘果式"和"播种式"拣货的区别，什么情况下该选择哪种策略？

### 核心答案

**订单波次（Wave）**：
将一批订单按一定规则（如相同配送路线、相同SKU、时效要求）合并为一个波次，统一分配库位、统一拣货，**目的是提高拣货效率和减少行走距离**。

**两种拣货模式对比**：

| 维度 | 摘果式（D2C，Decoction） | 播种式（D2S，Distribution to Sortation） |
|------|--------------------------|------------------------------------------|
| 流程 | 一个订单从多个库位取货，拣完即打包 | 多订单集中拣货，按订单分拨到各播种口 |
| 适用场景 | 订单量小、SKU多、精度要求高 | 订单相似度高、SKU相对集中 |
| 设备依赖 | 人工PDA为主 | 需电子标签（PTL）或自动化分拨线 |
| 效率 | 订单独立，切换频繁，效率低 | 集中拣货效率高，但二次分拨有损耗 |
| 精度 | 高（单人全流程负责） | 中（依赖分拨准确性） |

**策略选择建议**：
- 电商B2C小订单 → 播种式
- 工厂零部件、紧急订单 → 摘果式

**路径优化**：拣货路径通常采用S型或U型遍历，配合库位热力图将高频SKU预排在中段，减少无效行走。

### 追问方向
- 波次规则如何配置？是按SKU数量、订单时效还是配送路线来分组？
- 摘果式拣货中，如果某个库位库存不足，系统是跳过去还是停单等待？

### 避坑提示
- 混淆"摘果"和"播种"的核心逻辑是常见错误，需能用一句话概括本质差异。
- 不了解波次概念、直接说"一单一单拣"的，说明缺乏规模化运营思维。

## 6. 打包复核流程与称重校验

### 题目
打包复核环节主要复核哪些内容？称重校验在WMS中扮演什么角色？

### 核心答案

**复核内容**：
- **商品复核**：扫描复核（逐件扫码 vs 整单扫描），确保商品、数量、批次与出库单一致
- **包装复核**：商品规格与包装箱规格匹配（是否需要分箱、加固包装）
- **信息复核**：面单信息与订单一致（收件人、商品描述）

**称重校验**：
- 系统记录理论重量（商品净重+包材重量）
- 打包完成后过秤，系统比对**实际重量 vs 理论重量**
- 偏差超出阈值（通常±5%~10%）则触发预警，强制复核
- 目的：防止漏扫、错发、夹带，是出库质量的最后一道防线

**系统流程**：拣货 → 打包站报到 → 扫码复核 → 称重比对 → 异常拦截或放行 → 面单打印 → 交接

### 追问方向
- 称重偏差率设置多少合理？是固定值还是按商品单价动态调整？
- 复核时发现商品数量少了，系统如何处理？是否支持部分出库还是整单拦截？

### 避坑提示
- 不理解"称重校验"为何能兜住漏扫/错发问题的，说明对仓库质控理解浮于表面。
- 打包复核与配送承运商的交接责任划分不清晰的，容易在实务中产生纠纷。

## 7. 出库流程

### 题目
完整的出库流程包含哪些步骤？"挑库"环节的逻辑是什么？

### 核心答案

**出库全流程**：

```
订单接收 → 库存预占 → 波次分组 → 挑库分配 → 拣货 → 复核 → 打包 → 称重 → 交接出库 → 库存扣减
```

**挑库（Stock Allocation）分配逻辑**：

1. **库存预占**：订单进来后，系统先查可用库存并"占库"（冻结对应数量），防止超卖
2. **分配规则**：
   - 按**批次**（FIFO，近效期优先）
   - 按**库位**（拣货距离近优先）
   - 按**库存状态**（可用库存，剔除冻结/锁定）
   - 按**货主/库存类型**（优先分配同货主库存）
3. **挑库结果**：生成"挑库单"，记录每个SKU从哪个库位、哪个批次出多少数量

**交接环节**：
- 打印出库单/面单，与物流承运商交接
- 重要：交接签字后，库存正式扣减，责任转移至承运商

### 追问方向
- 订单预占后，库存冻结的时长如何管理？长时间不拣货是否自动解冻？
- 多货主场景下，挑库时跨货主分配库存是否允许？什么情况下会触发？

### 避坑提示
- 挑库不等于简单的"有库存就出"——没有批次策略和库位优先级概念的，说明缺乏精细化运营理解。
- 出库后库存扣减的时机（交接时 vs 打包时）不明确，会导致账实不符。

## 8. 盘点流程

### 题目
请描述WMS盘点的主要流程，并解释"初盘/复盘"和"盲盘"的概念。盘点差异如何处理？

### 核心答案

**盘点流程**：

```
盘点计划（定义范围、截止日期）→ 盘点任务下发 → PDA初盘 → 复盘（交叉互盘）→ 差异确认 → 审批 → 库存调整
```

**初盘 vs 复盘**：
- **初盘**：首次清点人员对自己负责的区域/库位进行清点，记录系统库存与实盘数量的差异
- **复盘**：由另一人或主管对初盘结果进行复核，通常针对差异较大的库位重点复核，防止人为误差

**盲盘（Blind Count）**：
- 盘点前**不显示系统库存数量**，盘点人员只知道库位码/商品码，不知道系统账有几件
- 盘点完毕录入实数，系统自动比对差异
- **目的**：防止盘点人员"凑数"（根据系统账来填实物），确保盘点真实性
- 大型仓库（ Walmart 配送中心）普遍要求盲盘

**差异处理**：

| 差异类型 | 处理方式 |
|----------|----------|
| 盘盈（实>账） | 需查明原因（收货未入库？调拨未确认？），经审批后增加库存 |
| 盘亏（实<账） | 同样需查明原因，审批后扣减库存，并记入"库存损耗"科目 |
| 差异率超标 | 触发专项复查，超出容差范围的须仓库经理签字 |

### 追问方向
- 盘点过程中，订单还在继续入库出库，系统如何处理（冻结库存还是动态盘点）？
- 盘点差异率有没有行业标准？超过多少算异常？

### 避坑提示
- 不知道"盲盘"概念的候选人，缺乏对数据真实性保障机制的理解，盘点质量难以保证。
- 盘点差异不设审批流、直接系统强平差异的，在合规审计中是严重缺陷。

## 9. 调拨流程

### 题目
调拨业务分为哪两类？请分别描述"库间调拨"和"库内移位"的系统处理差异。

### 核心答案

| 维度 | 库间调拨 | 库内移位 |
|------|----------|----------|
| **定义** | 仓库A → 仓库B（跨物理地点） | 同一仓库内库位A → 库位B |
| **触发场景** | 库存再平衡、备货调货 | 库位整理、归并批次、效期清理 |
| **流程** | 调拨申请 → 审核 → 源仓出库（减库存） → 运输 → 目标仓入库（加库存） | 移位申请 → 审核 → 源库位下架确认 → 目标库位上架确认 |
| **在途库存** | 有在途状态（调出方减库，调入方未收） | 无在途（实时下架+上架） |
| **物流属性** | 涉及运输，有交接单和承运商 | 纯内部操作，无需承运商 |
| **账务处理** | 可能触发内部交易价格（成本分摊） | 无财务影响，仅库存账务调整 |

**系统核心控制**：
- 调拨单必须经过审核流（大额调拨需总监审批）
- 移位时若源批次被订单预占，系统应禁止移位或提示替代方案

### 追问方向
- 库间调拨时，货物还在运输途中，调入方是否可以做入库确认？库存状态怎么定义？
- 移位申请发起后，系统是否支持批量移位（一次挪多个SKU）？

### 避坑提示
- 库间调拨和库内移位混淆的候选人，对仓库运营的复杂度认知不足。
- 调拨流程中没有"审核"概念的，在大型仓库多级审批场景下会出大问题。

## 10. 库位管理

### 题目
WMS中库位编码的规则是什么？库位的状态有哪些，分别代表什么含义？

### 核心答案

**库位编码规则**（常见方案）：

```
仓库代码 + 库区代码 + 货架排号 + 货架层号 + 库位序号
例如：A-01-03-02-05
A = A仓库；01 = 1号库区；03 = 3排货架；02 = 2层；05 = 5号格口
```

**规则设计原则**：
- **有规律可循**：编码即地址，新员工能"读码找货"
- **不重复**：全仓库唯一
- **留扩展位**：区域调整时编码体系尽量不变

**库位状态**：

| 状态 | 含义 | 业务说明 |
|------|------|----------|
| **可用（Available）** | 空置且可使用 | 新商品可直接上架 |
| **占用（Occupied）** | 有存货存放 | 商品在库，上架时可选择 |
| **锁定（Locked）** | 被订单预占/盘点锁定 | 不可上架，冻结状态 |
| **冻结（Frozen）** | 质检不合格/效期问题 | 不允许出入库 |
| **禁用（Disabled）** | 维护中/已废弃 | 永久不可用 |

### 追问方向
- 同一库位有多批次商品时，系统如何记录？是一个库位一条库存记录还是按批次分开？
- 库位编码重码了（比如新库区重新规划了编码体系），历史数据如何迁移？

### 避坑提示
- 库位编码"纯数字递增"（1,2,3...）而无层级语义的，无法指导实际找货，编码体系设计失败。
- 库位状态只有"空/满"两种的，缺乏对锁定/冻结等业务场景的理解。

## 11. 批次管理

### 题目
批次管理在WMS中如何实现？请说明生产日期、有效期、供应商批次与库存的关联关系。

### 核心答案

**批次管理的核心要素**：

| 字段 | 说明 |
|------|------|
| **批次号** | 系统自动生成或供应商提供，唯一标识一批货物 |
| **生产日期** | 制造商生产日期，决定效期计算起点 |
| **有效期/失效日期** | 固定有效期或按生产日期+保质期自动推算 |
| **供应商批次** | 供应商自己的批次号，用于追溯 |
| **入库时间** | 批次入库时间戳，用于FIFO排序 |
| **采购订单号** | 批次与采购入库的关联 |

**系统实现**：

1. **入库时**：系统要求输入/扫描批次信息（可手动、可对接供应商系统）
2. **存储时**：批次信息挂在库位+SKU上，记录批次+库位+数量
3. **出库时**：系统按FIFO推荐批次，挑库单显示批次号，拣货时扫描校验
4. **追溯时**：通过批次号可查到：哪个供应商、哪票采购、哪天入库、哪个库位

**多批次场景**：同一SKU多个批次同库位，系统按批次号+入库时间分开记账，出库时按批次维度扣减。

### 追问方向
- 如果供应商送货时没有批次号，系统如何处理？是强制要求还是允许后补？
- 批次效期计算，保质期天数是从生产日期起算还是从入库日期起算？

### 避坑提示
- 批次管理只是"记个批次号"而没有与出库策略联动的，等于没有批次管理。
- 不支持批次追溯（查不到供应商/采购单/质检报告）的，批次数据价值大打折扣。

## 12. 序列号管理

### 题目
序列号管理和批次管理的本质区别是什么？什么场景下需要用序列号管理？

### 核心答案

| 维度 | 批次管理 | 序列号管理 |
|------|----------|------------|
| 粒度 | 一批N件，统一管理 | 每件商品单独编号，独立追踪 |
| 精度 | 批次级别 | 件级别 |
| 适用场景 | 快消品、食品、药品箱/瓶 | 3C数码、汽车配件、高价值单品 |
| 成本 | 低（数据量小） | 高（需逐件扫码、存储量大） |
| 追溯能力 | 批次追溯 | 单件全生命周期追溯 |

**序列号管理场景**：
- **手机/电脑**：每台设备序列号唯一，保修期绑定、售后追溯
- **汽车零部件**： VIN码绑定，安全召回精准定位
- **医疗器械**：法规强制要求单品追溯

**系统实现**：
- 入库时逐件扫描序列号，系统记录：序列号+批次+库位+入库时间
- 出库时扫描序列号，系统校验是否为"目标序列号"（防错发）
- 序列号可关联保修期（保修起始=出厂日期或销售日期）
- 库存查询可按序列号维度查询当前位置和状态

### 追问方向
- 序列号库存在盘点时如何处理？是否需要逐件清点？
- 退货时序列号如何处理？是标记为"可用序列号"重新销售，还是直接报废？

### 避坑提示
- 候选人不清楚批次（Batch）和序列号（S/N）的使用边界，可能导致系统设计过度（什么都用序列号）或不足（高值商品没有序列号管理）。
- 序列号重复扫描（重复入库）的校验机制没有设计，是严重的系统漏洞。

## 13. 库存预警

### 题目
请解释安全库存、最低库存、呆滞料的定义，以及WMS中如何设置库存预警规则。

### 核心答案

| 概念 | 定义 | 计算逻辑/设置方式 |
|------|------|-------------------|
| **安全库存（Safety Stock）** | 防止意外需求波动和供应延迟的缓冲库存量 | 经验值或公式：`安全库存 = (最大日消耗量 × 最长供货周期) - 平均日消耗量 × 正常供货周期` |
| **最低库存（Min Stock）** | 触发补货或采购提醒的临界值 | 通常与安全库存持平或略低 |
| **呆滞料（Dead Stock）** | 长时间未使用/流动的物料（一般定义：>3~6个月无进出） | 按"最近一次移动日期"自动标记 |
| **库存预警** | 系统自动检测库存水位，触发补货提醒或采购单 | 设置公式：`预警 = 当前库存 ≤ 安全库存` |

**预警触发后的系统动作**：

| 等级 | 触发条件 | 系统动作 |
|------|----------|----------|
| 提示 | 库存 ≤ 最低库存 | 推送补货提醒 |
| 警告 | 库存 ≤ 安全库存 | 生成采购建议 |
| 紧急 | 库存 ≤ 0（断货） | 阻止出库、触发补货流程 |

**呆滞料处理**：系统自动推送呆滞料报表，人工判断：打折促销、退货供应商、报废。

### 追问方向
- 安全库存的公式里，供货周期日如何获取？是固定值还是动态的？
- 多货主场景下，安全库存是按仓库维度设置还是按货主维度设置？

### 避坑提示
- 安全库存和最低库存概念混淆的，导致预警触发时机不对，不是预警失效就是过度预警。
- 安全库存设置纯靠"拍脑袋"经验值，没有数据分析支撑的，预警效果难以保证。

## 14. 效期管理

### 题目
药品和食品行业的效期管理有哪些特殊要求？近效期预警、复检期、销毁流程分别是什么？

### 核心答案

**法规背景**：GSP（药品经营质量管理规范）要求药品经营企业实现效期全程可追溯。

**效期管理的核心概念**：

| 概念 | 说明 | 时间节点示例（以12个月保质期为例） |
|------|------|----------------------------------|
| **近效期预警** | 库存接近效期末尾，系统提前预警以便及时消化 | 效期前≤6个月开始预警 |
| **复检期** | 某些商品保质期虽过，但经检验合格可继续使用 | 效期后→复检合格→延长效期 |
| **销毁流程** | 效期届满或确认报废的商品，须经审批后系统做库存清零 | 效期当日或复检不合格 |

**系统实现**：
- 入库录入生产日期+效期，系统自动推算失效日期
- 出库强制FIFO + 近效期优先（效期最近的先出）
- 效期预警任务：定时扫描，生成近效期报表推送仓库主管
- 销毁流程：发起销毁申请 → 审批 → 系统扣减库存 → 记录销毁台账（法规要求保存记录）

**混批出库控制**：同一出库单若涉及多个批次，系统应优先推荐近效期批次，并记录出库批次用于追溯。

### 追问方向
- 近效期的标准是多少天？是固定值还是按商品类别区分？
- 效期管理中"冻结"和"锁定"的区别是什么？过效期商品是自动冻结还是需要人工操作？

### 避坑提示
- 药品/食品行业从业者若不了解GSP效期追溯要求的，系统设计存在合规风险。
- 销毁流程没有审批记录仅做系统删除的，在药监审计时会被质疑数据造假。

## 15. 货主管理（Tenant）

### 题目
WMS中货主（Tenant）的概念是什么？如何实现多货主的数据隔离和权限控制？

### 核心答案

**货主（Tenant）概念**：
同一个WMS系统中，服务多个货主（品牌方/供应商），各货主数据**物理或逻辑隔离**，各自管理自己的库存、订单、报表，互不可见。

**多货主隔离方案**：

| 隔离方式 | 实现 | 适用场景 |
|----------|------|----------|
| **数据库级隔离** | 每个货主独立数据库 | 数据量极大、合规要求极高（如医药） |
| **Schema隔离** | 同一库不同Schema | 中等规模、货主数量有限 |
| **逻辑隔离** | 同一表通过TenantID字段区分 | 货主数量多、数据量适中（主流方案） |

**数据权限控制**：
- **仓储维度**：用户只能看到有权限的仓库
- **货主维度**：用户登录后，系统自动过滤只看该货主的数据
- **功能维度**：不同角色（仓库管理员 vs 货主管理员）功能权限不同
- **数据范围**：库存查询、订单操作、报表查看均受货主ID约束

**典型场景**：3PL（第三方物流）服务商同时代理10个品牌，每个品牌作为独立货主，数据完全隔离。

### 追问方向
- 跨货主调拨时（比如品牌A库存不足，从品牌B借货），系统如何处理权限和账务？
- 货主管理员能否看到自己货主的操作日志？审计日志是否需要保留？

### 避坑提示
- 逻辑隔离（TenantID）如果没有强制行级过滤（所有查询加TenantID条件），容易出现数据泄露的bug。
- 多货主场景下系统性能（索引设计、查询优化）是高频翻车点，需要特别关注。

## 16. 计费模式

### 题目
WMS中常见的仓储计费模式有哪些？请说明仓储费、作业费、增值服务费的计费逻辑。

### 核心答案

**三大计费模块**：

| 费用类型 | 计费逻辑 | 常见计费方式 |
|----------|----------|--------------|
| **仓储费（Storage Fee）** | 按库存占用空间和时间收费 | 按库位体积（立方米）× 存储天数；或按托盘数 × 天数 |
| **作业费（Handling Fee）** | 按仓库操作频次收费 | 按单据张数（入库单/出库单）；或按件数/重量 |
| **增值服务费（VAS Fee）** | 额外增值服务 | 贴标/换包装/质检/加热/代发等，按次或按件计费 |

**仓储费细分**：

| 计费维度 | 说明 |
|----------|------|
| 正常库 | 标准费率 |
| 临期库/慢流库 | 加收（鼓励货主及时清库） |
| 整进整出 | 按入库批次整体计费，不逐件计费 |
| 免仓期 | 新入库有免仓优惠期（如30天免仓） |

**作业费细分**：
- 入库：卸货费 + 上架费（按件数或托盘）
- 出库：拣货费 + 复核费 + 打包费（按单或按件）
- 调拨：移位费（按件数）

**结算周期**：月结为主，WMS自动生成月结账单，包含各货主各费用明细。

### 追问方向
- 仓储费按体积计费时，商品的体积是系统自动计算还是手工录入？
- 货主对计费有异议时，系统支持哪些对账方式和账单调整机制？

### 避坑提示
- 作业费不区分操作复杂度（如重货vs轻货、整箱vs拆零）的，一刀切计费的方案在实务中争议很大。
- 计费规则没有版本管理的，费率调整后历史账单无法追溯，是常见的结算纠纷来源。

## 17. WMS与TMS集成

### 题目
WMS与TMS集成的主要场景有哪些？请描述出库通知和调度交接的集成逻辑。

### 核心答案

**集成背景**：WMS负责仓库内物流，TMS负责运输管理，出库环节两者必须无缝衔接。

**核心集成场景**：

| 场景 | WMS侧动作 | TMS侧动作 |
|------|-----------|-----------|
| **出库通知** | 出库单复核完成，触发"出库通知"给TMS（包含：收件人、件数、重量、体积、时效要求） | 接收通知，生成运单，安排车辆 |
| **调度交接** | TMS回传"车辆已到达"，WMS解锁装车环节；出库完成后WMS回传"已发货" | TMS更新运单状态为"已揽收" |
| **在途追踪** | WMS记录出库时间、交接人、承运商运单号 | TMS提供物流轨迹回传WMS |
| **签收回传** | TMS签收信息回传WMS，WMS关闭出库记录 | 完成结算闭环 |

**出库通知数据内容**：
```
出库单号 / 收件地址 / 商品明细（SKU+数量+批次） /
总重量 / 总体积 / 期望配送日期 / 承运商要求 / 温层要求（冷藏/常温）
```

**集成方式**：API实时推送 + 消息队列（MQ）异步解耦，避免系统间强耦合。

### 追问方向
- 如果TMS反馈车辆晚到，WMS的出库计划如何调整？系统是否支持动态排程？
- 承运商丢件/损毁后，WMS和TMS的责任边界是什么？系统如何支撑理赔？

### 避坑提示
- WMS和TMS没有集成，靠人工电话/微信传递信息的，是仓库数字化程度低下的典型表现。
- 出库通知数据不完整（比如没有重量体积）导致TMS无法报价排车的，是常见扯皮场景。

## 18. RFID/条码应用

### 题目
一维条码和二维条码在WMS中的适用场景有何不同？条码打印规则有哪些？

### 核心答案

**条码类型对比**：

| 类型 | 容量 | 典型应用 | 扫码方式 |
|------|------|----------|----------|
| **一维条码（EAN-13/UPC-A/CODE128）** | 20~30位数字/字母 | 商品码（EAN13）、库位码（CODE128） | 激光/红光扫描枪，线性扫描 |
| **二维条码（QR/DataMatrix）** | 可存储中文、URL、批量数据 | 箱码、批次码、追溯码 | 影像式扫描器，需摄像头 |
| **RFID** | 大容量，可群读 | 托盘级流转、整车盘点 | 读写器，无需对准 |

**条码打印规则**：

1. **码制选择**：
   - 商品销售包装 → EAN-13（唯一样品）
   - 库位标签 → CODE128（支持字母+数字+特殊字符，可变长度）
   - 批次追溯 → QR码（信息量大，可含生产日期+供应商+批次+有效期）

2. **打印内容规范**：
   - 清晰度 ≥ 300 DPI
   - 最小模块宽度 ≥ 0.264mm（1维条码）
   - 留白区（Quiet Zone）必须保证，否则扫码失败

3. **条码规则校验**：
   - EAN-13 最后一位为校验位，系统自动计算生成
   - 批次码必须唯一，不允许重复

### 追问方向
- 如果条码污损无法扫描，系统如何处理？是否允许手工输入？
- RFID标签和条码的成本差多少？在什么规模下值得投入RFID？

### 避坑提示
- 不了解EAN-13校验位算法的，产品经理在对接供应商条码时会踩大坑。
- 条码打印规则不规范（尤其是留白区和最小模块宽度），会导致仓库扫码失败率飙升。

## 19. AGV/自动化立体仓接口

### 题目
WMS与WCS（仓库控制系统）的交互协议主要有哪些？AGV调度指令的典型格式是什么？

### 核心答案

**WMS-WCS-设备层级架构**：

```
WMS（策略层） → WCS（设备控制层） → PLC/AGV/堆垛机（执行层）
```

**WMS ↔ WCS 交互协议**：

| 协议 | 特点 | 适用场景 |
|------|------|----------|
| **HTTP REST / SOAP** | 通用性强，部署简单 | AGV、独立设备 |
| **WebSocket** | 实时双向通信 | 高速流水线、实时状态反馈 |
| **MQTT** | 轻量级，IoT友好 | 大量传感器/AGV群控 |
| **OPC UA** | 工业级实时控制 | 立体库堆垛机、输送线 |

**典型AGV调度指令（WMS → WCS）**：

```json
{
  "task_id": "TK20231015001",
  "task_type": "PUTAWAY",       // PUTAWAY=上架， PICK=拣货， TRANSFER=移位
  "sku": "SKU-12345",
  "qty": 10,
  "source_location": "A-01-03-02-05",  // 当前托盘位置
  "target_location": "B-02-01-01-01",  // 目标库位
  "priority": 1,                 // 优先级 1=高，5=低
  "agv_type": "叉式AGV",
  "callback_url": "http://wms-api/callback/task_status"
}
```

**WCS → WMS 回告内容**：任务开始/完成/异常，设备当前位置，异常原因（碰撞、障碍物）。

### 追问方向
- AGV任务下发后，WMS是否需要等待WCS回告才能释放库位锁定？超时机制如何设计？
- 如果同一库位被AGV和人工同时操作，系统如何防冲突？

### 避坑提示
- WMS和WCS职责边界不清晰的（比如WMS直接控制PLC的），系统架构设计混乱，维护成本极高。
- AGV任务没有优先级和高可用策略的，高峰期会出现任务积压和死锁。

## 20. 报表统计

### 题目
WMS中最重要的报表有哪些？请说明库存台账、收发存报表、周转率如何计算。

### 核心答案

**核心报表列表**：

| 报表 | 核心内容 | 使用者 |
|------|----------|--------|
| **库存台账** | 实时库存（SKU+批次+库位+数量+状态） | 仓库主管、货主 |
| **收发存报表** | 期间内 入库/出库/结存 三栏账 | 财务、货主对账 |
| **周转率报表** | 库存周转天数和周转次数 | 管理层、货主 |
| **库位利用率** | 各库区/库位容积使用率 | 仓库规划 |
| **效期报表** | 近效期/呆滞料预警 | 仓库主管 |

**收发存报表格式**：

```
期初库存 + 本期入库 - 本期出库 = 期末库存
```

**库存周转率计算**：

| 指标 | 公式 | 说明 |
|------|------|------|
| **周转次数** | `周转次数 = 出库成本 ÷ 平均库存成本` | 越高说明流动性越好 |
| **周转天数** | `周转天数 = 360 ÷ 周转次数`（或 `平均库存 ÷ 日均出库量`） | 越少说明变现越快 |

> **平均库存** =（期初库存 + 期末库存）÷ 2  
> **日均出库量** = 期间总出库量 ÷ 期间天数

**收发存报表明细**：按月/季/年汇总，支持按货主、按仓库、按SKU分类筛选，是货主对账和财务结算的核心依据。

### 追问方向
- 收发存报表的"本期入库"是否包含调拨入库？还是只有采购/生产入库才算？
- 周转率为0（长期不动）的商品，系统中是否有自动标记呆滞料的机制？

### 避坑提示
- 库存台账实时性差（T+1甚至T+2才更新）的，对运营决策影响极大。
- 周转率公式用金额还是数量？不同货主/品类是否统一口径？口径不一会导致数据失真。

## 21. 复核打包规则

### 题目
打包复核时，商品关联规则和打包材配货的逻辑是什么？

### 核心答案

**商品关联规则（通常指"先入后出"的配套关系）**：

| 规则类型 | 说明 | 示例 |
|----------|------|------|
| **固定配套** | 多个SKU必须作为一个整体出库，不可拆单 | 整机+附件包、礼盒套装 |
| **比例配套** | 按固定比例搭配出库 | 买二赠一，配件1:2比例 |
| **互斥规则** | 某些SKU不可与另一些SKU同箱 | 食品+化学品不可同箱（合规要求） |

**打包材配货逻辑**：

| 维度 | 逻辑 |
|------|------|
| **根据商品规格匹配** | 系统预设"商品-推荐箱型"映射表（如：体积<0.01m³ → 1号箱） |
| **根据订单数量推算** | 多件商品自动计算所需最少箱数，避免浪费 |
| **包材库存关联** | 打包材作为独立SKU管理，配货时同步扣减包材库存 |
| **填充物规则** | 易碎品自动带气泡膜；液体自动加重包装 |

**复核流程中的防错机制**：
- 打包前：扫描箱码，确认箱型正确
- 打包中：逐件扫描，商品与面单逐行核验
- 打包后：称重比对，防止漏装/错装

### 追问方向
- 如果推荐箱型用完了，仓库人员是否可以自行换箱？换箱后系统如何记录和计费？
- 互斥规则是系统强控还是仅提示？强制控制的话，超规打包如何处理？

### 避坑提示
- 打包材配货如果不管包材库存，打包到一半发现包材没了，是仓库现场的重大事故。
- 商品关联规则没有系统强控、仅靠人工记忆的，高发搭配错漏。

## 22. 退货处理流程

### 题目
请描述RMA退货处理的全流程，以及退货验收和退货上架的系统处理逻辑。

### 核心答案

**RMA（Return Merchandise Authorization）退货处理全流程**：

```
客户申请退货 → RMA审核（是否符合退货条件） → RMA开单 → 退货到货 → 退货验收 → 不合格处理（销毁/退回） → 合格品退货上架 → 库存释放/退款
```

**退货验收环节**：

| 验收项目 | 判定标准 | 系统处理 |
|----------|----------|----------|
| 外观验收 | 商品是否完好、有无损坏 | 记录验收结果，拍照留存 |
| 数量验收 | 实际退货数量 vs RMA申请数量 | 差异登记，少收/多收处理 |
| 序列号验收 | 退货序列号是否在RMA范围内（防欺诈） | 扫描序列号，系统校验 |
| 效期验收 | 退货商品是否在有效期内 | 近效期/过期商品进入特殊处理流 |

**退货上架逻辑**：
- 验收合格 → 退货入库单 → 重新上架（同普通入库流程）
- 可选择**原库位上架**（退给原供应商）或**新品上架**（重新入库销售）
- 退货批次号继承原批次或生成新批次（取决于货主要求）

**不合格退货处理**：
- 外观损坏 → 残次品库位，不允许正常销售
- 过期 → 直接进入销毁流程，不计入可用库存

### 追问方向
- RMA退货验收费用的承担责任如何划分？验收不合格的退回运费谁承担？
- 跨年退货（原年度订单）是否允许？系统如何处理财务年度和库存年度的差异？

### 避坑提示
- 退货验收没有序列号校验的，是刷单欺诈的高发场景（用新货冒充退货）。
- 退货商品直接混回正品库存而不重新质检的，存在质量风险，尤其在医药/食品行业。

## 23. 虚入虚出

### 题目
什么是"虚入虚出"？委托代销和受托代销在WMS中如何处理库存和账务？

### 核心答案

**虚入虚出定义**：字面意思是"虚拟入库、虚拟出库"，本质是**货权不转移但货物实际在流转**的业务形态。所有权不变化的入库和出库，系统通过库存账务分离来实现。

**委托代销（货主把货发给经销商/渠道商代为销售）**：

| 维度 | 说明 |
|------|------|
| 库存归属 | 货权仍属于委托方（货主），代销方只管销售 |
| 入库（虚入） | 代销方WMS收到货，记"代销在库"（可用库存），不触发采购付款 |
| 出库（虚出） | 商品售出后，货权从委托方转移到消费者，代销方WMS记"代销出库"，触发结算 |
| 账务 | 定期对账，按实际销售量结算（卖多少结多少） |

**受托代销（经销商替货主代销）**：

| 维度 | 说明 |
|------|------|
| 库存归属 | 货权转移给受托方（经销商），受托方全权管理 |
| 入库（虚入） | 货到时不确认货权，只有"受托在库" |
| 出库（虚出） | 受托方将货再次销售后才确认货权转移 |
| 账务 | 受托方按委托方指定的结算价与委托方结算，高卖部分为受托方利润 |

**WMS系统实现要点**：
- 库存类型需要区分：`自有库存` / `代销库存` / `受托库存`
- 代销出库时，只有"代销库存"可用，自有库存不可用
- 虚入虚出单据与实入实出单据格式相同，但触发的账务逻辑不同

### 追问方向
- 委托代销的库存，盘点责任归谁？货主还是代销方？
- 虚入虚出的库存是否占用仓库容积？仓储费由谁承担？

### 避坑提示
- 虚入虚出业务如果WMS没有独立的库存类型区分，直接混在普通库存里管理的，盘点和结算都会乱成一锅粥。
- 代销/受托业务不区分的，财务对账时会出现大量"货发出去了但钱没收回来"的异常库存。

## 24. 库位容量管理

### 题目
库位容量管理包含哪些维度？货架层数和容器管理的逻辑是什么？

### 核心答案

**库位容量管理维度**：

| 管理维度 | 内容 | 系统字段 |
|----------|------|----------|
| **库位容积** | 长×宽×高，最大可存放体积 | 库位容积（m³） |
| **货架层数** | 每组货架有多少层，每层独立管理 | 货架层数，每层最大承重 |
| **容器管理** | 托盘/周转箱/料盒的规格和容量 | 容器类型、容器最大装载数 |
| **重量限制** | 每个库位的最大承重 | 库位承重（kg） |
| **面积限制** | 库位底面最大占用面积 | 库位面积（m²） |

**货架层数管理**：
- 货架编号+层号构成库位码（如 A-01-03-02，A仓库1号库区3排2层）
- 每层独立记录当前库存体积/重量
- 上架时系统校验：`当前库存体积 + 本品体积 ≤ 库位容积` 且 `当前重量 + 本品重量 ≤ 库位承重`

**容器管理**：
- 同一SKU在同一库位可能用不同容器（整托盘 vs 散箱）
- 容器规格定义：`容器类型 + 最大装载量（件数/重量）`
- 拣货时系统提示容器编号，作业人员按容器单元拣货（减少拆零）

**容量预警**：
- 库位使用率 > 80% → 预警（建议分流）
- 超过容积上限 → 禁止再上架该SKU

### 追问方向
- 异形商品（超长/超宽/超重）如何管理库位容量？是否有特殊的"非标准库位"类型？
- 库位容量是静态配置还是动态计算？随着商品规格变化如何更新？

### 避坑提示
- 库位容量只配一个总体积，没有拆分"可用体积/已占体积/预留体积"的，容量管理形同虚设。
- 货架层数没有独立重量限制的，重货放高层导致货架倾翻是严重安全隐患。

## 25. WMS权限设计

### 题目
WMS的权限体系通常分为哪几个维度？请说明仓库维度、货主维度、功能维度的权限控制设计。

### 核心答案

**三维度权限模型**：

| 维度 | 管控内容 | 实现方式 |
|------|----------|----------|
| **仓库维度** | 用户能操作哪些仓库 | 用户-仓库角色绑定，支持多仓库授权 |
| **货主维度** | 用户能看到/操作哪些货主的数据 | 数据行级过滤（TenantID），用户-货主角色绑定 |
| **功能维度** | 用户能做哪些操作（菜单/按钮级） | RBAC（基于角色的访问控制），菜单权限+按钮权限 |

**RBAC设计要点**：

```
用户 → 角色 → 权限
例：仓库管理员（角色） → 拥有 入库/出库/盘点/调拨 的增删改查权限
    货主管理员（角色） → 拥有 库存查看/报表/退货 的权限，无系统配置权限
```

**功能权限细化**：

| 权限粒度 | 说明 |
|----------|------|
| 菜单权限 | 可见哪些模块（如"报表"菜单是否可见） |
| 按钮权限 | 是否显示"审核通过"/"强制删除"等敏感按钮 |
| 字段权限 | 某些角色只能看字段A，看不到字段B（如成本字段） |
| 数据范围 | 超级管理员看全部仓库，普通仓管只看自己仓库 |

**多维度权限叠加**：
- 仓库A + 货主甲 + 功能：入库增删改查
- 仓库B + 货主乙 + 功能：仅库存查看
- 权限叠加原则：**取交集**（同时满足仓库、货主、功能条件才放行）

**特殊权限场景**：
- **数据隔离**：用户A操作仓库X的订单，绝对看不到仓库Y的订单（哪怕URL直接输入订单ID）
- **操作留痕**：所有敏感操作（删除库存记录、强制审核）均记录操作日志，含操作人+时间+IP

### 追问方向
- 仓库维度权限和货主维度权限哪个优先级更高？是否可能存在"仓库A的所有货主数据都能看"的超级管理员？
- 权限变更（如员工调岗）时，系统支持批量调整还是需要逐个用户修改？

### 避坑提示
- 权限设计只有"功能维度"而没有"数据维度"隔离的，是严重的数据安全漏洞——低权限用户可能通过API直接查询到无权数据。
- 权限变更后没有实时生效机制（需要重新登录才生效）的，在紧急人员调整时会非常被动。
